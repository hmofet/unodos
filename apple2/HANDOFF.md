# Apple II port — boot contract, harness ABI, and traps

This is the durable engineering reference for the UnoDOS/Apple II port:
the boot contract, the memory map, the harness ABI, and the non-obvious
traps discovered during bring-up, so they don't have to be re-derived.
The port is **shipped (M1–M3)** — desktop, RWTS read/write, the mini-FS,
the disk-loaded app set, audio, Theme and the scheduler verdict are all
done and harness-verified; the milestone-by-milestone *outcomes* (what
each app does, what was dropped, the OutLast/scheduler verdicts, the
WOZ/FloppyEmu imaging path) live in [README.md](README.md). What follows
is the lower-level material the README intentionally keeps out: the
exact boot/RWTS contract, the py65 harness internals, and the assembler
traps.

The port was built to the same bar as every other UnoDOS port (macplus
is the template): **code + a ROM-free regression harness + screenshots +
a README**, each milestone self-contained.

## 2. The Apple II envelope (what constrains every decision)

From the plan, restated as engineering facts:

- **CPU/RAM:** 6502 @ 1 MHz, 48–64 KB. No interrupts in use (no timer IRQ,
  no vblank IRQ on the base machine). The kernel is a **polling** main
  loop — there is no ISR layer like macplus's VIA/SCC handlers.
- **Video:** hi-res 280×192, effectively 1-bit. 7 pixels/byte, **bit 0 =
  leftmost** pixel, **bit 7 = palette** (keep it 0). Rows are
  **interleaved** (the notorious Apple II address scramble) — §4.
- **Input:** keyboard only as the standard device. `$C000` = key data
  (bit 7 set ⇒ key waiting, bits 6–0 = ASCII), `$C010` = clear strobe.
  No mouse at M1 (AppleMouse II card is backlog). So the M1 desktop is
  **keyboard-driven**: arrows / Return / Tab / ESC.
- **Sound:** 1-bit speaker click at `$C030` (toggle). Not needed for M1.
- **Storage:** Disk II, 140 KB GCR floppy, **no firmware sector services**.
  The boot ROM loads exactly track 0; our own RWTS (`boot.s`) does
  everything after. This is why the harness has to emulate the drive at
  the nibble level — §6.
- **No timer.** The Clock app uses a **calibrated soft tick**: count
  main-loop passes and convert to seconds with a hand-tuned constant.
  There is no hardware second.

**Target-machine floor.** Design M1 for the **Apple II+** (the floor):
left/right arrows only (`$08`/`$15`), no up/down, no Tab key in ROM. Keep
keyboard nav working with **left/right + Return + ESC** so it runs on a
II+; treat up/down/Tab as IIe niceties, not requirements. (Plan open
question #3 — confirm the FloppyEmu target machine when convenient. IIe
later unlocks up/down `$0B`/`$0A` and 80-col/aux memory.)

## 3. The `boot.s` contract — read this before kernel.s / mkdsk.py / harness.py

`boot.s` is the fixed interface all three new files must agree with. The
facts you must honor:

1. **ROM autoload.** The real Disk II P5A controller ROM loads boot sector
   T0S0 to `$0800` and jumps `$0801` with **X = slot×16** (slot 6 ⇒
   X = `$60`). `boot.s` byte 0 is `$10` = "autoload all 16 track-0 sectors
   to `$0800–$17FF`" (the standard multi-sector boot). So the *whole*
   4 KB track 0 is the boot image, not just one sector. **Risk:** a clone
   ROM that honors only 1 sector breaks this — first thing to check in
   AppleWin (documented in the plan). The harness should replicate the
   16-sector autoload directly (load `disk[0:4096]` → `$0800`).

2. **What boot0 does after autoload** (`entry:`): builds the 6-and-2
   decode table at `$1900`, then for tracks `1..KTRACKS` it `seek`s the
   head (stepper phase pulses on `$C080–$C087`) and `readsec`s every
   sector, denibbling each into `$4000 + 16*page` using the DOS 3.3
   `skew` table, then `jmp $4000`.

3. **Kernel load address = `$4000`** (`KERNLOAD`). Your `kernel.s` must
   `org $4000` and its entry point must be the first byte at `$4000`.

4. **`KTRACKS` placeholder.** At label `ktpatch` the code is
   `cmp #$4B` — the operand `$4B` ('K') is a placeholder. `mkdsk.py` must
   patch it to **(number-of-kernel-tracks + 1)** so the loop loads exactly
   the kernel's tracks. The kernel occupies tracks `1..KTRACKS`; the
   compare value is `KTRACKS+1` (loop runs while `zpTrkWant < KTRACKS+1`).
   Patch the operand of the unique two-byte sequence `C9 4B` (`CMP #$4B`).
   Assert exactly one occurrence, mirroring how `macplus/mkdisk.py`
   asserts on its `KSIZ` placeholder.

5. **Sector ordering.** `readsec` writes physical sector P to page
   `dst + skew[P]`, where
   `skew = $0,$7,$E,$6,$D,$5,$C,$4,$B,$3,$A,$2,$9,$1,$8,$F` (DOS 3.3 soft
   interleave). Consequence: the `.dsk` image is stored in **logical
   (DOS) order** — track-major, 16 logical sectors × 256 bytes each, in
   ascending logical order — and the **harness nibblizer** must place
   physical sector P's payload from **logical block `skew[P]`**. Get this
   wrong and it boots in AppleWin but not the harness (or vice-versa).
   This is the single most error-prone part — §6.

6. **The encode table is already here.** `boot.s`'s `wrtab` (the 64 valid
   "disk bytes") is exactly the 6-bit→nibble translation the harness's
   nibblizer needs to *write* GCR. Reuse it verbatim; don't reinvent it.

7. **Checksums are not verified on read** (trusted FloppyEmu/AppleWin
   media — see the comments). The harness must still emit *structurally*
   correct address/data fields (`D5 AA 96 …` / `D5 AA AD …`) because the
   RWTS hunts for those marks, but the data checksum nibble can be any
   valid nibble.

## 4. Memory map & the hi-res interleave (kernel ↔ harness must match)

Suggested M1 memory map (you own the machine after boot):

| Range | Use |
|---|---|
| `$0000–$00FF` | zero page (boot used `$F0+`; free to reuse post-boot) |
| `$0200–$03FF` | input buffer / vectors scratch |
| `$0800–$1FFF` | boot code + GCR scratch — **free to reclaim** once kernel runs |
| `$2000–$3FFF` | **hi-res page 1** = the display framebuffer |
| `$4000+` | **kernel code** (loaded by boot0). Stay below hi-res page 2 if you ever use it; for M1 use page 1 only, so `$4000` upward is yours. |
| above kernel | BSS / app buffers (Notepad, game boards, etc. at M2/M3), below `$C000` I/O |
| `$C000–$CFFF` | I/O soft switches (keyboard, speaker, Disk II, slot ROM) |

Pick a fixed `KBSS`-style base for zeroed buffers (macplus uses an
out-of-image `KBSS`; you can do the same, e.g. `$6000` or `$9000`), and
have the boot path / kernel entry clear it. Document the chosen base in a
comment so `mkdsk.py`'s "kernel fits" assert can reference it.

**Hi-res row address (the scramble).** For pixel row `y` (0..191) the
framebuffer byte address of column-byte `c` (0..39) on hi-res page 1 is:

```
addr = $2000 + (y & 7)*$400 + ((y >> 3) & 7)*$80 + (y >> 6)*$28 + c
```

Both the kernel renderer **and** the harness PNG de-interleaver must use
this exact formula. The kernel needs a fast row-address lookup (precompute
a 192-entry table of row base addresses at boot, or a 24-entry
"group base" + 8-entry "within-group" pair). 7 px/byte, bit 0 leftmost
(this matches the `font7.s` glyph convention from `mkfont.py`, so glyph
bytes can be ORed straight into the framebuffer at byte-aligned columns —
the "UnoDOS Lite" 7-px window-column constraint the plan calls out).

## 5. The harness ABI (UDM1 discovery header)

Copy the macplus convention exactly so the harness can find the kernel's
variables without hardcoding addresses. At the very top of `kernel.s`:

```
        org $4000
start:  jmp start2          ; 3 bytes
        .byte "UDM1"        ; magic the harness scans for
        .word vars          ; little-endian pointer to the var block
start2: ...
```

The harness, after boot, scans low RAM (or just reads `$4000`) for the
`UDM1` magic and reads the following word as the `vars` address. It then
reads **named fields at fixed offsets** from `vars` to assert kernel
state. In macplus the harness reads `mouse_x` at `vars+28`, `mouse_y` at
`vars+30`; you choose your own layout but keep the **first few fields
stable and documented** (suggest: `frame_ctr`, `sel_icon`, `zcount`,
`clock_secs`, `top_win`). The harness uses these to verify, e.g., that an
arrow key actually moved the selection or that the clock advanced.

`vars` is a labeled block of `dc.b`/`dc.w` at the end of `kernel.s`
(see `macplus/kernel.asm:2420` for the shape). Keep it in-image so it
loads with the kernel.

**Status:** the stub kernel already carries this header, and the harness
discovers it (magic at `$4003`, vars pointer at `$4007` — `_find_vars`
in harness.py). One stub-ism to remove: the stub places `vars` at
`org $5000`, padding the binary to 4098 bytes. The real kernel should
drop that `org` and let `vars` follow the code naturally — just keep the
whole image (code + vars) below `KBSS = $6000`, or raise `KBSS` (§7).

## 6. `harness.py` — built; status and remaining work

**Built and `--selftest` green.** The rig plays the Apple II ROM + Disk
II controller around a py65 6502 core. What exists (read the file,
~410 lines, before extending it):

- **ROM autoload:** `disk[0:4096]` → `$0800`, `X = $60` (slot 6),
  `PC = $0801` — exactly the P5A 16-sector autoload.
- **Disk II:** the data latch (`$C0EC`) serves a per-track GCR nibble
  stream built lazily by `nibblize_track` (sync `$FF` gaps, `D5 AA 96`
  4-and-4 address field with `vol^trk^sec` checksum, `D5 AA AD` +
  342+1-nibble 6-and-2 data field, DOS 3.3 `SKEW` interleave, volume
  `$FE`). **Key simplification, now a contract:** instead of modeling
  the 4-phase stepper, the harness reads the current track straight
  from zero page **`$F1`** (`zpTrkCur` in boot.s) — valid because
  boot.s's `seek` completes before any data-latch read, and the RWTS
  discards the other soft-switch values. **`$F1` is therefore ABI.**
  The M2 kernel gives itself its own write RWTS and keeps its
  current-track variable at `$F1` to preserve this contract (the
  alternative would be growing real phase stepping into the harness).
- **GCR codecs:** `enc44/dec44`, `nibblize_data/denibblize_data`,
  `nibblize_track/denibblize_track`, all derived as the mathematical
  inverse of boot.s's decode using boot.s's own `WRTAB`/`FLIP2`/`SKEW`
  tables, and proven by `--selftest` round trips (4-and-4 over all 256
  values; 6-and-2 over edge + random payloads; full tracks).
- **Keyboard:** `$C000` latch (bit 7 = key waiting), `$C010` clear on
  read or write; `key`/`keys` script commands; codes in `KEYS` (note:
  `left` = `$88` = the same code as `backspace` — that is real Apple II
  behavior, left-arrow IS backspace on a II+; design the kernel's nav
  accordingly).
- **Speaker `$C030`:** read returns 0, no-op.
- **Screenshots:** `shot NAME` de-interleaves hi-res page 1 with the §4
  formula → 280×192 grayscale PNG (`png_gray` lifted from macplus).
- **Failure detection:** executing `BRK` raises with the PC.
- **UDM1 discovery:** after each step batch it looks for `"UDM1"` at
  `$4003` and reads the vars pointer at `$4007`. It currently only
  *discovers* `vars` — nothing asserts against it yet (see d below).
- **Script commands:** `wait N` (N ticks of `TICK_INSTRS` MPU steps),
  `shot`, `key`, `keys`, `quit`; `--trace` prints PC/op/registers.

**Remaining harness-related work:**

a. **End-to-end stub proof (do this first).**

   ```sh
   dasm boot.s -f3 -obuild/boot.bin
   dasm kernel.s -f3 -obuild/kernel.bin
   python3 mkdsk.py build/boot.bin build/kernel.bin build/stub.dsk
   printf 'wait 500\nshot stub\nquit\n' | python3 harness.py build/stub.dsk shots
   ```

   Success = `shots/stub.png` shows the `$55` vertical-stripe pattern
   across the full 280×192. Budget instructions generously — boot0
   denibbles two tracks through a byte-level latch; if `wait 500` (1M
   steps) isn't enough, raise it (or loop until `$2000` reads `$55`).
   If it hangs hunting for marks, suspect the sync-gap or `SKEW`
   direction first (§11) and re-run with `--trace`.

b. **`TICK_INSTRS` calibration.** Currently a placeholder (2000). Once
   the real kernel's main loop + soft clock exist, pick the pair
   together: choose the kernel's passes-per-second constant, measure
   instructions per main-loop pass, and set `TICK_INSTRS` so `wait N`
   advances the soft clock predictably. Document both numbers in both
   files (§11).

c. **`$FBB3/$FBC0` seeds are wrong — fix when writing SysInfo.** The
   harness seeds both to `$E0`, which is not a real machine pair. Real
   values (Apple II Family Identification note): `$FBB3` = `$38` (II),
   `$EA` (II+), `$06` (IIe/IIc/IIGS); when `$FBB3=$06`, `$FBC0` = `$EA`
   (IIe), `$E0` (enhanced IIe), `$00` (IIc). M1 targets the II+ floor ⇒
   seed `$FBB3=$EA, $FBC0=$EA` in the harness and have SysInfo print
   "Apple II+". Keep the harness seed and SysInfo's decode table in
   lockstep. **The kernel must NOT call ROM routines** (no `COUT`, no
   monitor `WAIT`) — they don't exist in the ROM-free harness; it may
   only *read* these seeded identification bytes.

d. **(Optional) vars assertions.** If screenshots alone feel weak, add
   a script command (e.g. `expect OFF VAL` reading `vars+OFF`) — the
   macplus harness reads vars fields the same way. Screenshots stay the
   primary oracle either way.

## 7. `mkdsk.py` — done

Implemented per spec and exercised against the stub kernel: 143360-byte
DOS-order image, track 0 = `boot.bin` (asserts 4096 bytes + `$10` byte
0), kernel from track 1 zero-padded, the unique `C9 4B` patched to
KTRACKS+1, fits-below-`KBSS` assert with **`KBSS = $6000`**, one-line
summary print.

Two knobs you may need while growing the real kernel:

- **`KBSS = $6000` gives the kernel image only 8 KB** (`$4000–$5FFF`,
  code + vars). If the M1 kernel outgrows it, raise `KBSS` in
  `mkdsk.py` and move the kernel's BSS base to match — address space up
  to ~`$BFFF` is yours. Keep the two declarations (mkdsk.py constant,
  kernel.s BSS base) visibly in sync; a comment in each pointing at the
  other is enough.
- The harness consumes the same `.dsk` and nibblizes it (§6) — the
  *only* coupling between `mkdsk.py` and `harness.py` is "logical
  DOS-order 35-track image." Keep it that way.

## 8. `kernel.s` — the M1 desktop

`org $4000`, `processor 6502` (dasm syntax — see `boot.s` for the dialect:
`equ`, `dc.b`, `ds`, labels, `.byte` via `dc.b`). Mirror the macplus
*structure*, not its 68K code. M1 surface (plan §1):

1. **Boot/entry:** UDM1 header (§5); clear BSS; build the 192-entry hi-res
   row-address table; seed `vars`.
2. **Renderer primitives** (all byte-column / 7-px aligned — the honest
   "UnoDOS Lite" constraint):
   - `hgr_clear` (fill page 1),
   - `draw_char` / `draw_string` (OR `font7.s` glyph bytes at a byte
     column; the font is already in the right bit order),
   - `fill_rect` / `frame_rect` (byte-aligned x; this is why windows snap
     to 7-px columns),
   - optional 50% dither for the desktop background (Theme uses this at
     M3 — the macplus `pat_tab` model).
   `include "build/font7.s"` for the glyphs.
3. **Desktop + menu bar:** a top menu bar (`MENUBAR_H`), a dithered
   desktop, and a row/grid of app icons with labels (SysInfo, Clock at
   M1; the full 11-app roster lands across M2/M3).
4. **Window manager**, keyboard-driven: a `wintab` of up to `MAXWIN`
   windows (state/proc/x/y/w/h/title — see macplus `WENT_SIZE` layout), a
   `zlist` z-order, open / raise / close. Title bars drawn; the **focused
   (topmost)** window is visually distinct. No dragging at M1 (no mouse);
   window placement can cascade.
5. **Keyboard navigation** (II+ floor): on the desktop, **left/right**
   move the icon selection (`sel_icon`), **Return** launches the selected
   app, **ESC** closes the topmost window. When a window is focused, route
   keys to it; ESC stays kernel-side (closes). This is the macplus
   `handle_events` desktop/focus split (`macplus/kernel.asm:347`),
   re-expressed for `$C000` polling instead of an event queue — you can
   keep a tiny event queue or just poll-and-dispatch directly.
6. **SysInfo app:** detect the machine via `$FBB3`/`$FBC0` (harness-seeded)
   and render CPU/RAM/machine lines. Keep the strings short (40-col).
7. **Clock app:** a **calibrated soft tick** — increment a frame counter
   each main-loop pass; every N passes (N = your 1 MHz calibration
   constant) bump `clock_secs` and redraw HH:MM:SS. There is no RTC and no
   timer IRQ; this is the honest Apple II reality the plan calls out.
   Expose `clock_secs` in `vars` so the harness can assert it advances.
8. **Main loop:** poll `$C000`; if a key is ready, dispatch (and clear via
   `$C010`); else do periodic work (advance soft clock, refresh topmost).
   No ISRs. Keep it a single cooperative context (the M3 scheduler slots
   in here later, as in macplus `task_yield`).

Keep an `AUTOTEST` build switch (dasm `-DAUTOTEST=1`, test with
`ifconst AUTOTEST` / `ifnconst`) that auto-launches SysInfo + Clock at
boot, exactly like macplus — the regression script relies on it so it
doesn't have to navigate the launcher first.

## 11. Risks & open questions (carry these into the README / commit msg)

- **Autoload count (§3.1):** the `$10` 16-sector autoload assumes a
  16-sector P5A ROM. If a clone honors only 1 sector, fall back to
  replicating the ROM read loop inside `boot0`. First AppleWin check.
- **Skew/interleave (§3.5):** logical-vs-physical sector order is the
  classic "works in one emulator, not the other" trap. The harness
  selftest proves harness-internal consistency (nibblize → denibble
  round trip), but only AppleWin can prove the *absolute* convention
  matches real DOS-order `.dsk` semantics — flag it for the AppleWin
  pass.
- **`$F1` zpTrkCur ABI (§6):** the harness serves nibbles for whatever
  track `$F1` names. At M1 only boot.s touches the disk; the M2
  kernel-side write RWTS keeps its track variable at `$F1` to preserve
  this contract.
- **Soft-clock calibration (§6b):** the harness has no real time; keep
  `TICK_INSTRS` and the kernel's per-second pass-count in sync so
  screenshots are reproducible. Document both numbers.
- **`KBSS` headroom (§7):** 8 KB for the whole M1 kernel image is tight;
  raising `KBSS` is one constant in mkdsk.py plus the kernel's BSS base.
- **`$FBB3/$FBC0` seeds (§6c):** currently `$E0/$E0`, not a real machine
  pair — fix to II+ (`$EA/$EA`) alongside SysInfo's decode.
- **II+ vs IIe (plan Q#3):** M1 targets the II+ floor (left/right only;
  left = `$88` = backspace). Confirm the FloppyEmu target machine; IIe
  later unlocks up/down + 80-col.
- **OutLast at 1 MHz** was an M3 feasibility question; the verdict
  (ships at ~4 fps) is recorded in [README.md](README.md).
