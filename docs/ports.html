The UnoDOS family

pc64 is one world of many. The same GUI-first OS, and one machine-readable Contract, span 20+ hardware targets — from 8-bit consoles to modern ARM boards. Here is how the others differ.

One idea, many machines

Every port follows the same firmware-as-BIOS pattern pc64 uses with UEFI: ask the platform's firmware for a framebuffer and input, then let UnoDOS do the rest. The Raspberry Pi asks the VideoCore firmware for its framebuffer over the mailbox; the PowerPC Mac boots as an Open Firmware client; the consoles talk to their video chips and VRAM directly. A single Contract (unodef/) — including a shared logical window model — is what every world is generated from or checked against, so the ports stay consistent rather than diverging.

UI tiers

How much UI a port carries depends on how much memory it has:

  • Full desktop shell (unoui): the high-spec worlds — pc64, PlayStation 2, Dreamcast, and the ARM/PowerPC boards — can run the same themed toolkit desktop pc64 uses.
  • Minimal legacy path: the tiny targets — NES (2 KB RAM), Game Boy (8 KB), the C64 — keep a compact icon/button desktop; the full toolkit simply won't fit.

And where pc64 is written in C, the classic x86 kernel and the retro ports are hand-written in each CPU's native assembly — 6502, Z80, 68000, Sharp SM83, ARM7TDMI, NEC V30MZ, HuC6280, PowerPC and AArch64.
