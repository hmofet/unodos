# UnoDOS pc64 user manual (`docs/`)

This folder is the **user manual** for UnoDOS/pc64, published with GitHub Pages at
**https://hmofet.github.io/unodos/** (Settings → Pages → *Deploy from branch:
`master` / `/docs`*). It is a self-contained static site (`.nojekyll`), responsive
and light/dark aware. The repo's developer `*.md` files also live here and are
served as raw text; the site's own pages are the `*.html` files.

## How the site is built

The HTML is generated by [`build_site.py`](build_site.py). Page content lives in the
`PAGES` table in that script; one shared shell (sidebar + header + footer) wraps
every page.

```bash
python docs/build_site.py     # regenerates docs/*.html + assets/style.css + .nojekyll
```

Images live in [`assets/img/`](assets/img/) and are **not** produced by
`build_site.py` — they come from the capture tools below and are committed
alongside the HTML.

### House style (user-facing pages)

- Focus on **what UnoDOS does and how to use it**, not how it is implemented.
  Deep/internal detail belongs only in the Developer section (`dev-*.html`).
- **No em-dashes** anywhere (use commas / colons / periods; the page-title
  separator is `·`). Verify with a real Unicode check
  (`python -c "print(open(f).read().count(chr(0x2014)))"`), not Windows
  `grep -P` (its `\x{2014}` gives false negatives).

## Where the screenshots come from

| Images | Produced by | How |
|--------|-------------|-----|
| Desktop, themes, apps, browser, networking (`desktop.png`, `theme_*.png`, `app`/browser/network shots) | [`pc64/docs_shots.py`](../pc64/docs_shots.py) | Boots the real ESP under **QEMU + OVMF** headless and drives the desktop over QMP (keyboard-only; the emulated pointer doesn't reach the shell), dumping each scene to PNG. Run scenes by name, e.g. `python3 docs_shots.py themes editor browser_docs`. |
| The **Windows** flasher UI (`flasher-windows.png`) | [`pc64/flash/capture/`](../pc64/flash/capture/) | A scheduled task (registered once with highest privileges) launches the released flasher and captures its window with Win32 `PrintWindow`. |
| The **macOS** flasher UI (`flasher-macos.png`) | [`pc64/flash/capture/mac/`](../pc64/flash/capture/mac/) | A small app bundle (granted Screen Recording once) captures the flasher window in-process via ScreenCaptureKit and auto-trims it. |

Both flasher-capture pipelines download the **latest released** flasher, capture,
and never click Install (no drive is written). See their READMEs for the one-time
setup.

## The flashers

The USB flashers (what end users run) live in [`pc64/flash/`](../pc64/flash/):
`UnoDosFlash.cs` → `UnoDosFlasher.exe` (Windows, built on Windows + WSL) and
`flash/mac/` → `UnoDosFlasher.app` (macOS, built on a Mac). Both embed a gzipped
UEFI disk image made by `pc64/tools/mkuefi.py`.

They are **distributed via GitHub Releases**, not committed to the repo. Getting
started links to `releases/latest/download/UnoDosFlasher.exe` and
`...-macOS.zip`, so **every release must attach both assets with those exact
names** (and be a normal, non-prerelease release so `latest` resolves).

## Updating the manual when pc64 changes

1. Edit the relevant page content in `build_site.py` and run it to regenerate the HTML.
2. Regenerate any affected screenshots (`docs_shots.py` for the desktop; the
   `flash/capture` pipelines for the flasher UI) and copy the PNGs into `assets/img/`.
3. Rebuild the flashers, cut a new **GitHub Release** (`flasher-vX.Y.Z`) with both
   binaries, and push `docs/` to `master`. Pages redeploys automatically.
