Developer guide

For building UnoDOS pc64 from source, running it under QEMU, packing a bootable USB image, and building the flashers themselves. End users do not need any of this; the flasher covers them.

Toolchain

  • Build: x86_64-w64-mingw32-gcc (UEFI apps are PE32+ images, the mingw target's native output, so no gnu-efi or EDK2) and python3.
  • USB image: sgdisk (gptfdisk), mtools, python3.
  • Emulator: qemu-system-x86 + OVMF.

Build the image

From the pc64/ directory:

./build.sh            # build the unoui desktop shell -> build/esp/
./build.sh run        # build, then boot it in QEMU + OVMF

The build compiles the shell, the toolkit, the ten themes, the apps and the drivers, links a single BOOTX64.EFI, and stages a bootable ESP tree in build/esp/, a real FAT volume with the loader at EFI/BOOT/BOOTX64.EFI plus sample documents and the TrueType fonts. The default image is about 660 KB. A separate ./build.sh legacy builds the earlier icon-desktop shell, kept only as reference.

Run it in QEMU

./build.sh run launches QEMU with OVMF and the ESP as a virtual FAT disk:

qemu-system-x86_64 -machine q35 -m 256 \
  -drive if=pflash,format=raw,readonly=on,file=/usr/share/OVMF/OVMF_CODE_4M.fd \
  -drive if=pflash,format=raw,file=vars.fd \
  -drive format=raw,file=fat:rw:build/esp \
  -device qemu-xhci -device usb-tablet
Keyboard-firstThe whole desktop is driven from the keyboard, so QEMU needs no special input setup. QEMU is also natively scriptable over QMP (send-key + screendump); see the screenshot harness below.

Pack a real USB disk image

QEMU fakes a disk from the build/esp/ directory (-drive fat:rw:build/esp), but real UEFI firmware needs a partition table. tools/mkuefi.py turns the ESP tree into a real raw disk image: GPT with one EF00 EFI System Partition (FAT32) holding the whole tree:

./build.sh                    # -> build/esp/
python3 tools/mkuefi.py 512   # -> build/unodos-uefi.img  (512 MiB)

Boot-test the image directly by pointing QEMU's last drive at build/unodos-uefi.img instead of the ESP directory.

Build the flashers

The flashers (what end users run) each embed a single gzipped copy of unodos-uefi.img and stream it to the raw device. Source lives in pc64/flash/.

TargetBuild onCommandOutput
WindowsWindows + WSL (in-box csc.exe, no SDK)pc64\flash\build-flasher.ps1UnoDosFlasher.exe
macOSa Mac (Swift toolchain)pc64/flash/mac/build-app.shUnoDosFlasher.app (universal)

Both build scripts run build.sh → mkuefi.py → gzip → embed automatically; pass --skip-build to reuse an already-built build/unodos-uefi.img (handy when the pc64 image is built on another machine). Attach the binaries to a GitHub Release with pc64/flash/publish-release.ps1.

Cross-machine buildsBuilding the macOS .app needs the pc64 image, which requires x86_64-w64-mingw32-gcc. If your Mac lacks the mingw cross-compiler, build unodos-uefi.img on a Linux box and pass it to build-app.sh --skip-build.

Regenerate the manual screenshots

The screenshots in this manual are produced headlessly by pc64/docs_shots.py: it boots the real image under QEMU + OVMF and drives the desktop over QMP, capturing each scene to pc64/shots/manual/. Because OVMF doesn't surface the emulated pointer to the shell, every scene is driven from the keyboard.

python3 docs_shots.py                       # all scenes
python3 docs_shots.py themes editor browser_docs   # selected scenes
UNO_NIC=1 python3 docs_shots.py browser_http       # networking scenes (SLIRP)

Copy the PNGs you need into docs/assets/img/, then rebuild and commit. If the shell's app roster or the Control Panel's tab order changes, update the scene offsets in docs_shots.py first.
